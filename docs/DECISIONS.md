## Purpose

maintainability, data integrity, or learning outcomes. Do not record trivial

## D-001: Build an Offline-Only Application

**Status:** Accepted  
**Date:** 2026-07-17

### Context

The project is limited to one week and focuses on Flutter, GetX, and SQLite
fundamentals. Networking, authentication, and synchronization would introduce
unrelated complexity.

### Options Considered

1. Offline-only local application
2. Local application with a mock API
3. Application connected to a real backend

### Decision

The application will operate offline and use SQLite as its persisted source of
truth.

### Consequences

- The project remains focused on the intended fundamentals.
- Data is available only on the device.
- There is no account recovery or multi-device synchronization.
- Synchronization conflicts are outside the project scope.

### Reconsider When

A future learning objective explicitly requires networking, authentication, or
synchronization.

## D-002: Define Version-1 Food Identity and Storage Rules

**Status:** Accepted

**Date:** 2026-07-20

### Context

The version-1 `foods` table needs explicit rules for logical identity, numeric
validity, generated identifiers, and timestamps before its DDL is written.

Without these rules, the form, Dart model, DAO, and SQLite constraints
could accept different definitions of valid food data.

### Options Considered

#### Name Identity

1. Treat names with different casing as different foods.
2. Treat names case-insensitively while preserving display casing.

#### Duplicate Names

1. Allow duplicate logical names.
2. Require names to be unique after trimming and case normalization.

#### Calories

1. Require calories to be greater than zero.
2. Allow zero while rejecting negative and non-finite values.

#### Identifiers

1. Generate identifiers in the form or controller.
2. Use SQLite `INTEGER PRIMARY KEY` generation.
3. Use SQLite `INTEGER PRIMARY KEY AUTOINCREMENT`.

#### Timestamps

1. Store normalized ISO-8601 text.
2. Store UTC epoch seconds as integers.
3. Store UTC epoch milliseconds as integers.

### Decision

The version-1 food schema will use these rules:

- Food names are required and trimmed before persistence.
- Names containing only whitespace are invalid.
- Name comparison and uniqueness are case-insensitive.
- Display casing is preserved; `Apple` may be stored and displayed as entered.
- Duplicate logical names such as `Apple` and `apple` are not allowed.
- Calories are required, finite, and greater than or equal to zero.
- SQLite enforces non-null and non-negative calorie constraints.
- Dart validation additionally rejects non-finite values such as `NaN` and
  `Infinity`.
- Drift generates identifiers using its `autoIncrement()` column declaration,
  which produces SQLite `INTEGER PRIMARY KEY AUTOINCREMENT`.
- `created_at` and `updated_at` are required Drift `DateTimeColumn` values.
  Application writes normalize them to UTC before persistence.

### Rationale

- Case-insensitive uniqueness prevents accidental duplicates while keeping the
  user-facing name readable.
- Allowing zero supports valid zero-calorie items.
- Drift-generated identifiers provide atomic uniqueness without assigning
  persistence identity to the UI. The current `autoIncrement()` declaration is
  accepted for this learning project even though its never-reuse behavior is
  not a product requirement.
- Typed `DateTimeColumn` mapping keeps conversion details out of feature code;
  UTC writes make range comparisons and ordering deterministic.

### Consequences

- Application input must be trimmed before insertion.
- Database constraints remain necessary even though the form validates input.
- Case-insensitive comparison uses SQLite's built-in behavior, whose default
  `NOCASE` support is primarily ASCII-oriented.
- The simple uniqueness rule cannot represent two foods with the same name but
  different serving definitions. Until serving metadata exists, names must be
  descriptive, for example `Apple (100 g)` and `Apple (1 piece)`.
- DAO inserts must return the identifier generated by SQLite.
- Timestamp writes must consistently normalize `DateTime` values to UTC.
- Calendar-day queries must use explicit UTC range boundaries derived from the
  desired local day.
- Raw SQL that interprets Drift timestamp storage requires deliberate conversion
  and must not assume a unit without verifying the configured Drift mapping.

### Reconsider When

Revisit this decision when:

- Food serving units become first-class fields.
- Unicode-aware case folding is required.
- Foods are imported from multiple catalogs with external identifiers.
- Identifier reuse or insertion overhead becomes a measured concern.
- A backend or synchronization contract mandates another timestamp format.

## D-003: Use Drift as the SQLite Access Layer

**Status:** Accepted

**Date:** 2026-07-20

### Context

The project needs to practise SQLite constraints, relationships, queries, and
migrations while still integrating cleanly with Dart. Direct `sqflite` calls
would expose raw maps and migration code, while Drift adds typed tables, query
composition, generated DAOs, and schema verification.

### Options Considered

1. Use `sqflite` and handwritten SQL throughout.
2. Use Drift tables and DAOs, with custom SQL where it improves learning.
3. Hide SQLite behind a generic persistence abstraction.

### Decision

Use Drift for the database connection, schema declarations, DAOs, generated
models, and migration tooling. Continue learning SQL directly through
constraints, joins, aggregates, custom queries, and query-plan inspection.

### Rationale

- Typed generated APIs reduce accidental mapping errors.
- Versioned schema snapshots make migration behavior testable.
- Drift still permits custom SQL and direct SQLite inspection where those are
  the learning objective.
- A generic repository layer would add indirection without a second persistence
  implementation to justify it.

### Consequences

- Generated files and schema snapshots are part of the development workflow.
- Table or DAO changes require code generation.
- Schema changes require a version increment, generated migration scaffolding,
  deliberate migration code, and tests.
- The developer must understand which behavior comes from SQLite and which is a
  Drift abstraction.

### Reconsider When

Revisit this decision if a future exercise specifically requires raw `sqflite`
mapping or if Drift prevents access to required SQLite behavior.

## D-004: Preserve Meal History with Snapshots and Soft Deletion

**Status:** Accepted

**Date:** 2026-07-21

### Context

Changing or deleting a food must not silently rewrite the meaning of an existing
meal log. Daily totals need the values that were recorded at consumption time,
not the food's latest values.

### Options Considered

1. Read the current food name and calories through a required foreign key.
2. Cascade-delete meal entries when a food is deleted.
3. Store historical values on each meal entry and make the food link optional.

### Decision

Each meal entry stores `food_name_snapshot` and
`calories_per_serving_snapshot`. Its nullable `food_id` uses `ON DELETE SET
NULL`. Ordinary food removal is represented by nullable `foods.deleted_at`, and
active-food queries exclude those rows.

### Rationale

- Historical totals remain stable after a food is edited.
- Meal history survives physical deletion of its source food.
- Soft deletion supports removal from the active catalog without immediately
  destroying relational context.

### Consequences

- Snapshot values intentionally duplicate food data.
- Aggregate queries must calculate from meal-entry snapshots.
- Restoring a soft-deleted food and handling duplicate names need deliberate
  behavior because name uniqueness still applies to deleted rows.
- Physical-delete and soft-delete behavior both require focused tests.

### Reconsider When

Revisit this decision if audit requirements demand immutable food revisions or
if food restoration becomes part of the product scope.

## D-005: Treat APK Updates as a Database Compatibility Contract

**Status:** Accepted

**Date:** 2026-07-22

### Context

The target distribution workflow sends APK files directly instead of relying on
the Play Store. Android retains application data only when the update is
installed over the same application identity and is accepted as an upgrade.

### Options Considered

1. Verify migrations only with in-memory database tests.
2. Add emulator testing with debug builds.
3. Combine automated migration tests with signed release APK upgrade tests.

### Decision

For meaningful schema changes, keep automated Drift migration tests and also
test a representative release upgrade by installing the new APK over the old
one without uninstalling or clearing data. Release builds must keep the same
application ID and signing certificate and increase the Android `versionCode`.

### Rationale

Automated tests validate schema paths and data integrity quickly. The release
upgrade exercise additionally validates Android package identity, signing,
retained app data, and first-open migration behavior.

### Consequences

- Released schema snapshots and migration steps are forward-compatibility
  artifacts and must not be rewritten casually.
- The signing key is operationally required for future updates and must be
  backed up securely outside source control.
- Losing the signing key prevents an in-place update under the same application
  identity.
- A successful APK installation alone is insufficient; persisted records must
  be checked after the upgraded app opens its database.

### Reconsider When

Revisit this decision if distribution moves to a managed store or device
management system with a different signing and rollout contract.
